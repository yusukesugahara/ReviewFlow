# ドメインモデル

## Tenant
契約組織を表す。すべての業務データは Tenant に属する。

主な責務:
- 契約組織の識別
- プラン管理
- テナント状態管理

## User
テナントに所属する利用者。

主な責務:
- 認証主体
- テナントロール保持
- 申請者 / 承認者 / 管理者として操作するときの主体

申請者・承認者は `User.role` だけでは決まらない。申請者は `Application.applicant_user_id`、承認者は `ApprovalStep.assignee_user_ids`、スペース内の管理権限は `GroupMember.role` で決まる。

## Invitation
メンバー招待を表す。**テナント管理者（tenant_admin）** が、自テナントにユーザーを追加するために発行する。

主な責務:
- 招待メール相当の管理
- 招待状態管理
- 招待時ロール保持
- 招待者（invited_by）としてテナント管理者を記録する想定

## Group
UI 上のスペースを表す。backend 実装では `groups` として扱い、フォーム、承認フロー、申請、CSV出力ジョブの業務境界になる。

主な責務:
- スペースの識別
- テナント内での業務データ分離
- スペース参加者の所属先

## GroupMember
ユーザーのスペース参加を表す。

主な責務:
- スペース所属の管理
- スペースごとの `admin` / `user` ロール保持
- 同じユーザーがスペースごとに異なる権限を持つ状態の表現

## FormDefinition
申請フォーム定義の親。

主な責務:
- フォームの識別
- 公開状態管理
- 対象テナント内でのフォーム種別管理
- 同一スペース内の複数フォーム管理
- 公開申請URLでは `formDefinitionId` を指定し、申請者アクセストークンに利用フォームを固定する

## FormField
フォームの1項目。

主な責務:
- 項目名
- 項目タイプ
- 必須設定
- 選択肢
- 表示順

## ApprovalFlow
フォームに紐づく承認フロー定義。

主な責務:
- 対象フォームに対する承認ルール管理
- ステップ集合の管理

## ApprovalStep
承認フロー上の1ステップ。

主な責務:
- 承認順序
- 承認担当者の割り当て
- 差し戻し可否

承認担当者はロール名ではなく、同一スペース所属ユーザーのID配列で管理する。1ステップに複数人を割り当てられるが、MVPの承認経路自体は直列で進む。

## Application
申請実体。

主な責務:
- フォームに基づく申請の作成
- ステータス管理
- 現在承認ステップ管理

申請は作成時点の `form_definition_id` と `approval_flow_id` を保持し、後からフォーム定義や承認フローが更新されても、どの定義で提出された申請かを追跡できるようにする。

## ApplicationFieldValue
申請ごとの入力値。

主な責務:
- フォーム項目ごとの値保持

## ApplicationApproval
承認処理履歴。

主な責務:
- 誰がいつ承認 / 差し戻し / 却下したか保持

## CorrectionRequest
差し戻しイベント。

主な責務:
- 差し戻し全体コメント
- 修正依頼状態管理

## CorrectionRequestItem
差し戻し対象フィールド。

主な責務:
- 修正対象フィールドの識別
- 項目別コメント保持

## ExportJob
CSV出力ジョブ。

主な責務:
- 非同期出力管理
- ファイル生成状態管理

## AuditLog
監査ログ。

主な責務:
- 重要操作の証跡保持
