# ReviewFlow

## 概要
ReviewFlow は、組織ごとに独立したワークスペースを持ち、各組織が自分たち専用の申請フォーム・承認フロー・申請データ・監査ログを管理できる、マルチテナント対応の申請ワークフローSaaSである。

本プロダクトは、差し戻しや再提出が多い申請業務を対象に、組織内ユーザーだけでなく **組織外の申請者** も想定する。テナントのアカウント登録をせず、メールで届く申請者用リンク（アクセストークン）から申請・差し戻し修正・再提出まで行えることを価値とする。

本プロダクトは、上記を含む申請業務を対象に、以下を一元管理する。

- 申請フォーム定義
- 承認フロー定義
- 申請作成 / 提出 / 再提出
- 修正依頼付き差し戻し
- 承認 / 却下
- CSV出力
- 監査ログ
- テナントごとのユーザー管理（**テナント管理者によるユーザー招待**を含む）
- **使用状況ダッシュボード**（`/space`、テナント管理者向け）

## 管理画面（テナント向け）
契約組織の **テナント管理者（tenant_admin）** が参照する画面。自組織ワークスペース内の利用状況を把握するためのものであり、プラットフォーム運営者専用のコンソールとは別とする。

実装では、申請件数・平均差し戻し数・再提出件数などの指標は **`/space` ダッシュボード** に表示する。`/admin` はスペース管理・ユーザー招待・監査ログへの入口であり、使用状況の集計画面ではない。

表示する指標の例:

- **申請件数**（期間内など、集計単位は実装時に定義）
- **平均差し戻し数**（例: 申請あたりの差し戻し回数の平均）
- **再申請（再提出）件数**（再提出に至った件数・回数など、定義は実装時に明文化）

## 解決したい課題
既存の申請業務では、以下が分断されがちである。

- フォーム入力
- 差し戻し連絡
- 修正箇所の指摘
- 再提出
- 承認進捗の可視化
- 集計 / CSV出力

特に、差し戻し時に「どの項目を直すべきか」が曖昧で、メールや電話で補完しているケースを改善する。

組織外の申請者にとっては、差し戻しのたびにアカウント発行や社内システムへのログインを求めると手続きが止まりやすい。ReviewFlow では、案内メールや差し戻し通知メールに含まれる URL から `/apply/form` や `/apply/correction` にアクセスし、**テナント未登録のまま** 修正対象項目の確認と再提出ができる。

## コア機能
- マルチテナント
- ロールベース認可
- 動的フォーム定義
- 承認フロー定義
- 修正依頼付き差し戻し
- **公開申請**（申請者アクセストークンによるログイン不要の申請・差し戻し修正・再提出）
- 非同期CSV出力
- 監査ログ
- テナント向けダッシュボード（`/space` で使用状況を可視化）

## MVPスコープ
- テナント作成（サインアップ時にテナントと初回 `tenant_admin` を同時作成）
- テナント管理者作成
- メンバー招待（**テナント管理者（tenant_admin）** が自テナントにユーザーを招待し、承認者・申請者などを追加できる）
- パスワードリセット
- ロール管理
- フォーム定義 / 公開
- 承認フロー定義
- 申請作成 / 提出
- 公開申請（組織外申請者向け。案内メール → `/apply/form`、差し戻し → `/apply/correction`）
- 承認 / 差し戻し / 再提出 / 却下
- 修正対象フィールドの指定
- CSV非同期出力
- 監査ログ
- 使用状況ダッシュボード（`/space` で申請件数・平均差し戻し数・再申請数などを表示）
