# ビジネス要件

ReviewFlow は、差し戻し前提の申請業務を、フォーム定義・承認フロー定義・進捗管理・修正依頼・再提出まで一元管理できるSaaSである。

## ターゲット
- 小規模〜中規模組織
- Excel / メール / 紙運用が残っている組織
- 申請種別ごとに項目や承認ルールが異なる組織
- **組織外の申請者**（取引先、補助金申請者、施設利用者など）を含む手続き。相手にテナントアカウントを発行せず、メールリンクだけで申請から差し戻し対応まで完結させたい組織

## 提供価値
- テナントごとに独立運用できる
- 申請フォームを設定で増やせる
- フォームごとに承認フローを変えられる
- 差し戻し時に修正箇所を明示できる
- 修正が必要な項目だけ再入力させられる
- **組織外の申請者が、テナントアカウントなしで申請・差し戻し修正・再提出できる**（申請案内メールや差し戻し通知メールの URL と申請者アクセストークンで本人確認）
- 進捗と履歴を一元管理できる
- 重要操作を監査ログに残し、申請状態変更だけでなくユーザー招待、ユーザー削除・復活、スペース変更、スペースメンバー変更も後から確認できる
- ログイン中ユーザーが自分のプロフィール、メールアドレス、パスワードを管理できる
- スペース別の申請フォーム管理画面と申請一覧で、公開状態・申請状況・CSV 出力を確認できる

## ロール
- tenant_admin … テナント管理者。自テナントのユーザー招待、ユーザー管理、スペース管理、監査ログ確認を行う。
- tenant_user … テナントユーザー。所属スペースで申請作成や割り当てられた承認を行う。

スペースロールは `User.role` ではなく、スペース参加 (`group_members`) ごとに保持する。したがって同じユーザーが A スペースでは `admin`（スペース管理者）、B スペースでは `user`（スペースユーザー）になることがある。

申請者・承認者は `User.role` では表現しない。申請者は申請レコードの `applicant_user_id`、承認者は承認フローの各ステップに割り当てられたユーザーで決まる。

### 外部申請者（公開申請）

組織外の申請者は、テナントの `tenant_user` アカウントを持たない。代わりに次の流れで操作する。

1. 管理者が申請案内メールを送信する（`POST /form-definitions/groups/:groupId/request-access`）
2. 申請者がメール内 URL から `/apply/access` を開き、申請者アクセストークンを取得する
3. `/apply/form` で申請を入力・提出する
4. 差し戻し時は通知メールの URL から `/apply/correction` を開き、修正対象項目だけ編集して再提出する

認証は JWT ログインではなく `ApplicantAccessGuard`（`X-Applicant-Access-Token`）で行う。申請レコードの `applicant_user_id` が `null` の場合は、テナント未登録の外部申請者による申請を表す。

詳細は [フロントエンド設計](07_frontend_design.md) の `/apply/*` ルート、[API 仕様](04_api_spec.md) の `public/applications` 系、[差し戻し機能](10_correction_feature.md) を参照する。

## 招待
- **テナント管理者** が、同一テナントに参加させたいユーザーを招待する（メールアドレス、付与ロール、任意で初期スペース `groupId` とスペースロールを指定可能）。
- 招待の受諾・本登録フローは認証・メンバー管理の画面/APIで扱う。

## 重要な制約
- 1契約 = 1テナント（1テナントに複数スペースを持てる）
- ユーザーの複数テナント所属はMVPでは未対応
- 承認フローは直列承認のみ
- 承認者は承認ステップごとのユーザー割り当てを基本とする
- 高度な条件分岐フォームはMVP対象外
- 公開申請者の本人確認は申請者アクセストークンとメール受信を前提とする（テナントのパスワードログインとは別系統）
