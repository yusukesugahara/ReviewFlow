## 2. アプリ概要

**ReviewFlow** は、外部ユーザーからの申請受付に対応した、申請・承認ワークフロー管理アプリです。

特徴は、**ログインアカウントを持たない外部ユーザーでも申請できること**と、従来のフォーム送信だけでは扱いにくい**差し戻し・再提出までを一連の申請フローとして管理できること**です。

市役所・自治体における補助金申請、施設利用申請、各種届出など、住民や事業者からの申請受付を主な利用例として想定しています。  

申請者はアカウント作成の手間なく申請を提出できます。職員・管理者側では、申請内容の確認、承認、差し戻し、再提出内容の確認、CSV出力、監査ログ確認までを一元管理できます。

単なるフォーム送信ではなく、申請後の承認、差し戻し、再提出、履歴確認までを扱える申請アプリとして開発しました。

## 3. 開発背景・市場分析

申請業務をデジタル化する手段としては、Googleフォーム、専用フォーム作成ツール、ワークフローSaaS、Excel、メール、紙など、さまざまな選択肢があります。

しかし、既存のフォームツールは「申請を受け付ける」ことには向いていても、申請内容に不備があった場合に、**どの項目を修正してほしいのかを明確に伝え、申請者が修正して再提出し、最終的に承認まで進める**という一連の流れまでは扱いにくいと感じました。

実際の申請業務では、最初の提出だけで完了するとは限りません。  
記入漏れ、添付書類の不足、金額や内容の確認などにより、職員や承認者が申請者に修正を依頼し、申請者が内容を直して再提出する場面が発生します。

そこで ReviewFlow では、単なるフォーム送信アプリではなく、**差し戻しと再提出を前提に、申請を完成させていくための申請・承認ワークフロー管理アプリ**として設計しました。

また、自治体やバックオフィスの申請業務では、入力フォーム、承認依頼、差し戻し連絡、再提出、集計、履歴確認が別々のツールに分かれやすい課題があります。  
特に小規模な部署や一時的な申請業務では、専用システムを導入するほどではない一方で、メールやExcel、紙で補完すると確認状況や履歴が分散しやすくなります。

ReviewFlow では、申請フォーム、承認フロー、差し戻し内容、再提出、申請状態、CSV出力、監査ログを同じ業務フロー上で扱える構成にし、申請受付から承認完了までを一元管理できるようにしました。

### 解決したい課題

実務の申請業務では、以下のような課題が起きやすいと考え、設計に反映しました。

#### フォーム送信だけでは申請が完結しない

申請内容に不備があった場合、修正依頼や再提出が必要になります。  
しかし、一般的なフォームツールでは、提出後の差し戻し、修正、再提出までを一連の業務フローとして扱いにくい場合があります。

#### 差し戻し時の修正指示が曖昧になりやすい

差し戻し理由がメールや口頭で補足されるだけだと、どの項目を修正すべきか分かりにくくなります。  
その結果、修正漏れや再提出時の手戻りが発生しやすくなります。

#### 申請手段と承認手段が分散する

Googleフォーム、Excel、メール、紙などを組み合わせて運用すると、申請内容、承認状況、差し戻し内容、再提出履歴が別々の場所に分かれやすくなります。

#### 現在の対応者が分かりにくい

誰が次に対応すべきか、現在どの承認ステップにあるかが共有されにくく、確認や催促が属人化しやすくなります。

#### 操作履歴の追跡が難しい

誰がいつ申請、承認、差し戻し、却下を行ったかを後から確認しづらい場合、問い合わせ対応や内部確認に時間がかかります。

#### 運用管理が属人化しやすい

CSV出力や監査ログが整備されていない場合、集計や確認のたびに個別対応が必要になり、担当者に運用ノウハウが偏りやすくなります。

これらの課題をまとめて扱うため、ReviewFlow では、申請フォーム定義、承認フロー定義、申請状態、差し戻し情報、再提出履歴、監査ログを同一システム上で管理する構成を採用しました。

## 4. デザイン方針・UI の意図

ReviewFlow の UI は、自治体や業務部門の担当者、住民・事業者などの申請者が、迷わず目的の操作を行えることを重視して設計しました。

特に意識したのは、**今何を確認すべきか、次に何をすればよいかが分かりやすいこと**です。  
申請業務では、申請者、承認者、管理者で必要な情報や操作が異なります。そのため、画面ごとの役割を明確にし、ユーザーが迷わずに申請、確認、承認、差し戻し、再提出を行える構成を目指しました。

### デザインで優先したこと

#### やりたい操作にすぐたどり着けること

申請者、承認者、管理者それぞれが、画面を見たときに「自分が何をすればよいか」が分かることを重視しました。

申請者には申請入力、修正対象、再提出の操作を中心に表示し、承認者には申請内容、承認状況、差し戻し履歴、承認・差し戻しボタンを中心に配置しています。  
管理者には、申請フォーム、承認フロー、提出済み申請、監査ログなど、管理対象ごとに画面を分けて表示しています。

#### 画面ごとの役割を明確にすること

1つの画面に多くの機能を詰め込みすぎると、どこで何をすればよいか分かりにくくなります。

そのため、画面ごとに役割を分け、操作の目的が混ざらないようにしました。

- `/space`：スペース全体の利用状況を確認する画面
- `/space/[spaceId]/applications`：申請フォームを管理する画面
- `/space/[spaceId]/submissions`：提出済み申請を確認する画面
- `/admin/audit-logs`：監査ログを確認する画面

画面の責務を分けることで、ユーザーが「どの画面で何をするのか」を判断しやすくしています。

#### 情報を詰め込みすぎず、読み取りやすくすること

申請一覧、申請詳細、監査ログなどは情報量が多くなりやすいため、余白を広めに取り、カード、テーブル、セクションの区切りを分かりやすくしました。

特に業務画面では、見た目の派手さよりも、繰り返し使ったときに疲れにくく、必要な情報をすぐ見つけられることが重要だと考えています。

#### 状態を見ただけで判断しやすくすること

申請業務では、現在の状態をすぐに把握できることが重要です。

そのため、申請中、承認済み、差し戻し、却下などの状態をラベルで表示し、一覧画面や詳細画面で現在の状態が分かるようにしています。  
色は状態理解の補助として使用し、色だけに依存せず、テキストでも状態を確認できるようにしています。

#### 差し戻し内容を分かりやすくすること

ReviewFlow では、従来のフォーム送信だけでは扱いにくい、差し戻しと再提出の流れを重視しています。

そのため、差し戻し時には「どの項目を修正してほしいのか」が分かるように、差し戻し対象の項目とコメントを申請データに紐づけています。  
申請者が再提出する際にも、修正すべき内容を確認しやすくし、修正漏れや手戻りを減らせるように設計しました。

#### 外部ユーザーでも迷わず申請できること

申請者はログインアカウントを持たない外部ユーザーであることを想定しています。

そのため、申請画面では業務システムに慣れていない人でも、入力から提出まで進めやすいことを意識しました。  
アカウント作成を不要にし、申請に必要な情報入力に集中できるようにすることで、住民や事業者などの申請者側の利用ハードルを下げることを目指しています。

### 成果指標として見ていること

本アプリはポートフォリオ制作のため、実運用での利用継続率や申請数はまだ計測していません。

一方で、実際に運用する場合は、単に機能があるかどうかではなく、ユーザーが迷わず使えているか、継続的に業務で利用されているかを確認する必要があると考えています。

| 指標 | 見たい理由 |
| --- | --- |
| 利用継続率 | 一度使った部署や担当者が、継続して申請業務に使っているかを見るため |
| 申請フォーム作成数 | 管理者が複数の手続きに展開できているかを見るため |
| 申請数 | 外部ユーザーからの申請受付に実際に使われているかを見るため |
| 差し戻し後の再提出率 | 差し戻し後に、申請者が迷わず修正・再提出できているかを見るため |
| 差し戻し回数 | 修正指示が分かりやすく、手戻りを減らせているかを見るため |
| アクティブユーザー数 | 管理者、申請者、承認者がそれぞれ継続的に利用しているかを見るため |

## 5. 想定ユーザー・利用シーン

### 想定ユーザー

ReviewFlow では、申請を受け付ける側と、申請する側の両方を想定しています。  
特に、住民・事業者・取引先など、**ログインアカウントを持たない外部ユーザーからの申請受付**に対応している点を特徴としています。

| 役割 | 主な利用者 | 主な操作 |
| --- | --- | --- |
| **管理者** | テナント管理者、組織・部署の管理担当者 | スペース管理、ユーザー管理、申請フォーム作成、承認フロー設定、監査ログ確認、使用状況の把握 |
| **申請者** | 住民、事業者、取引先、社内ユーザーなど | 申請入力、提出、差し戻し内容の確認、修正後の再提出 |
| **承認者** | 職員、管理者、承認ステップに割り当てられた担当者 | 申請内容の確認、承認、却下、差し戻し |

申請者は、必ずしもログインアカウントを持つユーザーである必要はありません。  
外部申請者向けには、メールURLなどから申請・再提出できる導線を想定し、アカウント作成の手間なく申請できる構成にしています。

一方で、管理者・承認者側は、申請フォームや承認フローの管理、申請内容の確認、差し戻し、監査ログ確認など、継続的な業務管理を行う利用者として想定しています。

承認者は専用の固定ロールだけで判定するのではなく、申請の状態や承認フロー上の担当者設定に応じて、操作権限が変わる設計にしています。  
これにより、「現在の承認ステップに割り当てられている担当者だけが承認できる」といった、実務に近い権限管理を表現しています。

### 利用シーンの例

ReviewFlow は、単にフォームを送信するだけではなく、申請後に内容を確認し、必要に応じて差し戻し、申請者が修正して再提出する業務を想定しています。

主な利用シーンは以下です。

- 補助金申請
- 施設利用申請
- 各種届出
- 変更届・申請書類の提出
- 事業者からの許可申請・登録申請
- その他、複数人で確認・承認が必要な手続き

特に、市役所・自治体における住民や事業者からの申請受付のように、**組織外の申請者と内部の承認者が関わる手続き**に向いた構成にしています。

いずれの利用シーンでも、以下の流れを共通パターンとして扱えることを想定しています。

```text
申請フォーム作成
↓
外部ユーザーまたは社内ユーザーによる申請提出
↓
承認者による内容確認
↓
承認 / 却下 / 差し戻し
↓
差し戻し内容の確認
↓
申請者による修正・再提出
↓
最終承認
↓
CSV出力・監査ログ確認
```

## 6. 主な機能

ReviewFlow では、申請を受け付ける管理者、申請を提出する申請者、申請内容を確認する承認者の3つの役割を想定して機能を整理しています。

### 管理者向け

#### 申請フォーム作成

スペース単位で申請フォームを作成できます。  
申請内容に応じてフィールドを追加・並べ替えでき、作成したフォームは公開・アーカイブできます。

#### 承認フロー設定

申請フォームに対して、承認ステップや担当者を設定できます。  
複数人での確認や、段階的な承認が必要な申請業務を想定しています。

#### 使用状況ダッシュボード

申請件数、平均差し戻し数、再提出件数などを確認できます。  
申請業務がどの程度利用されているか、どこで差し戻しが多いかを把握できるようにしています。

#### 申請一覧確認

スペース内の提出済み申請を一覧で確認できます。  
ステータスや日付などをもとに、現在どの申請が対応中なのかを把握できます。

#### CSV出力

申請データをCSVとして出力できます。  
集計や内部確認、他システムでの利用を想定し、CSVエクスポートジョブを作成して、完了後にダウンロードできる構成にしています。

#### 監査ログ確認

申請、承認、差し戻し、ユーザー招待、権限変更、申請フォーム作成・編集などの操作履歴を確認できます。  
誰が、いつ、何を行ったかを確認できるようにし、問い合わせ対応や内部確認に使える設計にしています。

#### スペース・ユーザー管理

スペースの作成・削除、メンバー追加、テナントユーザー招待、ロール変更に対応しています。  
部署や業務単位で申請フォームや承認フローを分けて管理できるようにしています。

---

### 申請者向け

#### 申請フォーム入力

申請フォームに設定された項目に沿って、申請内容を入力できます。  
住民・事業者・取引先などの外部ユーザーでも使いやすいよう、必要な項目を順に入力できる構成にしています。

#### ログイン不要での申請提出

申請者は、ログインアカウントを持っていなくても申請を提出できます。  
外部ユーザーからの申請受付を想定し、アカウント作成の手間なく申請できる点を特徴としています。

#### 差し戻し内容の確認

申請が差し戻された場合、差し戻し対象の項目、項目別コメント、全体コメント、差し戻し履歴を確認できます。  
どこを修正すればよいかが分かるようにし、修正漏れや再提出時の手戻りを減らすことを意識しています。

#### 修正後の再提出

差し戻し後、申請者は修正内容を反映して再提出できます。  
単なるフォーム送信ではなく、不備のある申請を修正し、承認に進めるための流れを扱えるようにしています。

#### 申請案内メールの利用

公開申請URLの案内メール送信に対応しています。  
外部申請者に対して、申請ページへの導線を案内できる構成にしています。

---

### 承認者向け

#### 申請内容確認

申請者が提出した内容、承認進捗、過去の差し戻し履歴を確認できます。  
現在の申請状態や、過去にどのような修正依頼があったかを見ながら判断できます。

#### 承認

現在の承認ステップに割り当てられた担当者が、申請内容を確認して承認できます。  
承認フローに沿って、次のステップまたは完了状態へ進めます。

#### 却下

審査中の申請を却下できます。  
承認できない申請について、承認フローを終了させる操作として扱います。

#### 差し戻し

申請内容に不備がある場合、フィールド単位で差し戻し対象を選び、コメント付きで申請者に修正依頼できます。  
「どの項目を、なぜ修正してほしいのか」を明確に伝えられるようにしています。

#### 修正依頼内容の確認

差し戻し時に指定した項目やコメントを、申請詳細画面で確認できます。  
承認者側も、過去の修正依頼や再提出内容を確認しながら審査を進められます。

## 8. 設計上のアピールポイント

### 8-1. 状態遷移を持つ業務ワークフロー

ReviewFlow は、申請データを登録・更新するだけの単純な CRUD アプリではなく、申請の状態に応じて操作可能な内容が変わる業務ワークフローとして設計しています。

申請は、下書き、審査中、差し戻し、承認済み、却下といった状態を持ち、提出、承認、差し戻し、再提出、却下の操作に応じて状態が遷移します。

特に重視したのは、従来のフォーム送信だけでは扱いにくい「差し戻し」と「再提出」の流れです。  
申請内容に不備があった場合、承認者が修正対象を指定して差し戻し、申請者が内容を修正して再提出できるようにしています。

また、審査中の申請では現在の承認ステップを保持し、承認フロー定義と組み合わせて「今誰が対応すべきか」を判断できる構成にしています。

状態遷移のルールは `ApplicationTransitionPolicy` に集約し、サービス層から直接ステータスを書き換えないようにすることで、状態変更のルールが分散しないようにしました。

---

### 8-2. 実務を想定した権限管理

ReviewFlow では、単純な管理者・一般ユーザーというロールだけではなく、申請の状態や承認フロー上の担当者設定に応じて操作権限を判定しています。

たとえば、承認操作は、審査中の申請に対して、現在の承認ステップに割り当てられた担当者、またはテナント管理者のみが実行できるようにしています。

`ApplicationAccessPolicy` では、主に以下の条件を組み合わせて閲覧・操作権限を判定しています。

- テナント管理者かどうか
- 申請者本人かどうか
- 現在の承認ステップの担当者かどうか
- 過去に承認操作へ関与したユーザーかどうか
- 対象スペースにアクセスできるユーザーかどうか

フロントエンドでも操作ボタンの表示制御を行いますが、最終的な権限判定はバックエンド側で行う構成にしています。  
これにより、画面上でボタンを非表示にするだけでなく、API レベルでも不正な操作を防げるようにしています。

また、スペース単位の利用権限は `SpaceAccessService` で判定し、API ガードには `JwtAuthGuard`、`RolesGuard`、`ApplicantAccessGuard` などを組み合わせています。  
ログイン済みユーザーだけでなく、外部申請者向けのアクセスも分けて扱えるようにしている点が特徴です。

---

### 8-3. SaaS を想定したマルチテナント設計

ReviewFlow は、複数の組織や部署が同じアプリケーションを利用することを想定し、テナント単位・スペース単位でデータを分離しています。

サインアップ時にはテナントと初期管理者を作成し、認証情報にも `tenantId` を含めることで、ユーザーが所属するテナントの範囲内で操作できるようにしています。

データ管理の単位は、主に以下の2つです。

| 単位 | 役割 |
| --- | --- |
| テナント | 組織全体の境界。ユーザー、監査ログ、テナント全体の管理単位 |
| スペース | 部署・業務単位の境界。申請フォーム、承認フロー、申請データ、メンバー管理の単位 |

UI 上では「スペース」として扱い、API や一部のプロパティでは後方互換のため `groupId` として扱っています。  
概念としては、部署や業務単位で申請フォームや承認フローを分けるための管理単位です。

この構成により、同じアプリケーション上で複数の部署・業務を扱いつつ、申請データや承認フローが混在しないようにしています。

---

### 8-4. 監査ログによる操作履歴の管理

業務システムでは、単に処理が完了するだけでなく、後から「誰が・いつ・何をしたか」を確認できることが重要です。

ReviewFlow では、申請、承認、差し戻し、却下、ユーザー招待、権限変更、申請フォーム作成・編集などの操作を監査ログとして記録しています。

監査ログでは、主に以下の情報を扱います。

- 操作日時
- 操作者
- 操作内容
- 対象データ
- 操作結果
- 関連するスペース
- 実行者ロール
- HTTP メタデータ

管理者向けの監査ログ画面では、操作履歴を一覧で確認でき、期間、キーワード、結果、リスクなどで絞り込めるようにしています。

また、権限変更やユーザー招待、申請フォーム作成・編集など、影響範囲が大きい操作は高リスクとして分類し、確認しやすいようにしています。

| リスク | 主な対象 |
| --- | --- |
| 高リスク | ユーザー権限変更、ユーザー招待、申請フォーム作成・編集 |
| 要確認 | 操作失敗、データ出力、変更・削除操作 |
| 通常 | 参照系の操作など |

これにより、問い合わせ対応や内部確認が必要になった場合でも、操作履歴をもとに状況を追跡できる構成にしています。

---

### 8-5. OpenAPI による型安全な API 連携

ReviewFlow では、フロントエンドとバックエンドの連携ミスを減らすため、OpenAPI を用いた型安全な API 連携を採用しています。

バックエンドでは NestJS の Swagger 連携により OpenAPI スキーマを生成し、API 仕様を管理しています。  
フロントエンドでは `openapi-typescript` により TypeScript 型を生成し、`openapi-fetch` のクライアントに適用しています。

これにより、API のパス、リクエスト、レスポンスの型ずれをコンパイル時に検知しやすくしています。

```typescript
openapi-typescript ../backend/schema.json -o ./src/lib/api-schema.d.ts
```

また、画面実装側で扱いやすいように、ドメインごとの型を整理して再エクスポートしています。

フロントエンドとバックエンドの契約を OpenAPI に寄せることで、API 仕様の変更に気づきやすくし、保守性を高めることを目指しました。

---

### 8-6. バックエンドの責務分離

バックエンドでは、Controller や Service に業務ルールを詰め込みすぎないよう、責務ごとにクラスやサービスを分けています。

特に申請領域では、権限判定、状態遷移、入力値検証、提出処理、承認処理、差し戻し処理を分離し、変更影響を把握しやすい構成を意識しました。

| クラス / モジュール | 役割 |
| --- | --- |
| `ApplicationAccessPolicy` | 申請の閲覧・レビュー操作が許可されるかを判定 |
| `ApplicationTransitionPolicy` | 申請状態の前提チェックと状態遷移 |
| `ApplicationFormValueValidator` | フォーム入力値が定義どおりかを検証 |
| `ApplicationSubmissionService` | 提出・再提出処理の流れを管理 |
| `ApplicationReviewActionService` | 承認・却下・差し戻し処理の流れを管理 |
| `ApplicationCorrectionService` | 差し戻し対象や修正依頼に関する処理 |
| Repository 層 | 永続化処理を担当 |

この分離により、仕様変更が発生した場合に、権限、状態遷移、入力検証、永続化のどこを修正すべきかを判断しやすくしています。

たとえば、差し戻し条件を変更する場合は差し戻し関連のサービス、ステータス遷移のルールを変更する場合は `ApplicationTransitionPolicy` を中心に確認できるため、Service が肥大化しにくい構成を目指しています。

また、Repository 層を独立させることで、サービス層から DB 操作の詳細を切り離し、ビジネスロジックと永続化処理の責務を分けています。

## 9. 開発で意識したこと

ReviewFlow では、画面を作ることだけでなく、実際の申請業務で使われることを想定して、業務フロー・権限・状態管理・保守性を意識して開発しました。

### 業務運用を前提に設計すること

単に申請フォームを作成して送信するだけではなく、申請後に発生する確認、差し戻し、修正、再提出、履歴確認までを一連の業務として扱えるようにしました。

特に、差し戻し対象を項目単位で管理し、申請者が「どこを修正すればよいか」を確認できるようにした点を重視しています。  
また、CSV出力や監査ログなど、実際の運用で必要になりやすい機能も設計に含めました。

### ログイン不要の外部申請者を想定したこと

ReviewFlow では、管理者・承認者だけでなく、住民・事業者・取引先などの外部申請者も利用することを想定しています。

そのため、申請者がログインアカウントを持っていなくても申請できる導線を用意し、アカウント作成の手間なく申請を提出できる構成にしました。

外部ユーザーが使う画面では、業務システムに慣れていない人でも迷わず入力・提出・再提出ができることを意識しています。

### 権限、状態遷移、監査ログ、データ分離を重視したこと

申請業務では、誰が申請したか、誰が承認すべきか、現在どの状態にあるかが重要になります。

そのため、単一のロール判定だけに頼らず、申請状態、申請者本人かどうか、現在の承認ステップの担当者かどうかを組み合わせて操作可否を判定する設計にしました。

また、複数組織・複数部署での利用を想定し、テナント・スペース単位でデータを分離しています。  
主要な操作は監査ログとして記録し、後から「誰が・いつ・何をしたか」を確認できるようにしました。

### バックエンドでは責務分離を意識したこと

バックエンドでは、Controller や Service にすべての処理を詰め込まず、業務ルールの置き場所を分けることを意識しました。

権限判定は Policy、入力値検証は Validator、提出・承認・差し戻しの流れは Service、永続化処理は Repository に分けることで、仕様変更時にどこを修正すべきかを把握しやすくしています。

特に、状態遷移や認可ルールは変更が入りやすい部分のため、独立した単位として扱えるようにしました。

### フロントエンドでは迷わず操作できる構成を意識したこと

フロントエンドでは、見た目の装飾性よりも、ユーザーが迷わず目的の操作を行えることを重視しました。

再利用する UI は `src/components/` に配置し、画面固有の部品は各ルートの `_components/` に配置する方針を取っています。  
申請表示、フォームビルダー、承認フロービルダーなど、ドメイン単位でコンポーネントを分けることで、画面の責務を分かりやすくしました。

### OpenAPI を利用してフロントエンドとバックエンドの整合性を保つこと

フロントエンドとバックエンドの連携では、OpenAPI を利用して API 仕様を管理しています。

バックエンドの API 仕様から TypeScript 型を生成し、フロントエンド側で利用することで、リクエストやレスポンスの型ずれを減らす構成にしました。

API仕様を単一のソースとして扱うことで、画面実装とバックエンド実装の認識違いを減らすことを目指しています。

### テスト可能な単位を確保すること

状態遷移や認可ルールは、業務ロジックの中でも不具合が起きると影響が大きい部分です。

そのため、Policy や Validator など、単体で検証しやすい単位に分けることを意識しました。  
また、主要 API についてはバックエンド E2E テストでも確認できるようにし、申請提出、承認、差し戻し、再提出といった重要な流れを検証しやすい構成にしています。

## 10. PDCA・今後の改善予定

実装状況を確認したうえで、未実装または十分ではない領域を整理しています。

### 改善サイクルの考え方

今後は「画面を追加する」だけでなく、利用状況の指標から継続的に使われているかを確認し、導線・フォーム作成体験・申請体験を改善する想定です。


| Check              | Action                         |
| ------------------ | ------------------------------ |
| 利用継続率が低い部署やスペースを確認 | 初回設定、申請フォーム作成、承認フロー設定の導線を見直す   |
| 申請フォーム作成数が伸びない     | テンプレート、複製、既存承認フローの再利用 UI を改善する |
| 申請数が伸びない           | 公開申請 URL、案内メール、申請フォームの入力体験を見直す |
| アクティブユーザー数が偏っている   | 申請者・承認者・管理者それぞれの画面導線や通知を改善する   |


### 機能面

- **ファイル添付**  
申請フォームにファイル型フィールドやアップロード API は未実装です。現状はテキスト中心のフィールド構成です。
- **承認フローの分岐条件**  
現状は直線的な多段承認が中心で、金額や項目値に応じた分岐条件は未対応です。
- **既存承認フローの再利用 UI**  
`ApprovalFlowSelector` コンポーネントは存在しますが、画面への統合は未完了です。
- **メール通知の運用強化**  
招待・申請案内・差し戻しなどのメール送信機能は実装済みですが、本番向け SMTP 設定や通知テンプレートの拡充、送信失敗時の運用設計は今後の課題です。

### 品質・テスト

- **フロントエンド E2E テストの拡充**  
Playwright は導入済みですが、現状は認証クッキー検証など最小限のシナリオです。申請〜承認〜差し戻しまでの E2E は今後追加予定です。
- **権限管理ロジックのさらなる共通化**  
Policy / Service への分離は進めていますが、ロール判定の重複整理は継続的な改善項目です。


### 運用・プロダクト

- **本番環境へのデプロイ**  
Docker Compose によるローカル開発環境と CI は整備していますが、本番デプロイ構成はこれから整備する予定です。
- **UI/UX 改善**  
業務フローは一通り実装していますが、一覧画面の操作性やモバイル表示など、実利用を想定した改善余地があります。
- **操作ログ・監査ログの検索性向上**  
日本語ラベル表示、リスク分類、高リスク絞り込みは実装済みです。操作内容での全文検索やエクスポート、200件を超える履歴のサーバー側リスクフィルタなど、運用向けの分析機能は今後の拡張候補です。
