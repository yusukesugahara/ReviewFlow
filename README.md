# ReviewFlow

ReviewFlow は、市役所・自治体における補助金申請、施設利用申請、各種届出などを想定した、申請・承認ワークフロー管理アプリです。

申請フォームの作成、外部ユーザーからの申請受付、承認、差し戻し、再提出、CSV 出力、監査ログ確認までを一連の業務フローとして管理できます。

このアプリで特に重視したのは、単に「フォームを作って送信できること」ではありません。
実際の申請業務で起きる、確認、不備の指摘、修正依頼、再提出、承認履歴の確認まで含めて、業務が止まらず進むことを重視して設計しました。

私は、業務システムを開発するうえで、画面の見た目や機能数だけでなく、利用者が迷わず使えること、運用後に確認しやすいこと、変更に強い構成であることを大切にしています。

ReviewFlow ではその考え方をもとに、以下の点を意識して開発しました。

* ログインアカウントを持たない外部ユーザーでも申請できること
* 申請後に、誰が確認し、どこで止まり、何を直すべきか分かること
* 差し戻し内容を項目単位で明確にし、再提出まで追跡できること
* 権限、状態遷移、監査ログを整理し、実務に近い業務フローとして扱えること
* フロントエンドとバックエンドの責務を分け、保守しやすい構成にすること

本ドキュメントは、転職活動・技術紹介向けのポートフォリオ説明です。
業務要件、API 仕様、状態遷移などの詳細は `docs/` を参照してください。

## 2. アプリ概要

ReviewFlow は、住民、事業者、取引先など、組織外の申請者からの申請受付を想定した申請・承認ワークフロー管理アプリです。

一般的なフォームツールは、申請を受け付ける用途には便利です。
しかし、実際の申請業務では、提出後に不備が見つかり、承認者が修正を依頼し、申請者が内容を直して再提出する場面が発生します。

そのときに、申請内容、差し戻し理由、修正対象、再提出履歴、承認状況が別々のツールに分かれていると、確認漏れや手戻りが起きやすくなります。

ReviewFlow では、申請の提出だけでなく、提出後の確認、差し戻し、再提出、承認、履歴確認までを一連の業務フローとして扱えるようにしました。

主な流れは以下です。

1. 管理者が申請フォームを作成する
2. 外部ユーザーまたは社内ユーザーが申請を提出する
3. 承認者が申請内容を確認する
4. 必要に応じて承認・却下・差し戻しを行う
5. 申請者が差し戻し内容を確認し、修正して再提出する
6. 最終承認後、CSV 出力や監査ログで履歴を確認する

単なるフォーム送信ではなく、申請後に「誰が確認し、どこで止まり、何を修正し、どのように承認されたのか」まで追跡できる点を重視しています。

## 3. 開発背景・課題設定

申請業務をデジタル化する手段としては、Google フォーム、Excel、メール、紙、専用ワークフローシステムなどがあります。

ただ、実際の業務では、フォームで受け付けるだけでは完結しないケースがあります。
たとえば、記入漏れ、添付書類の不足、金額の確認、申請内容の修正などにより、承認者が申請者に差し戻しを行い、申請者が修正して再提出する必要があります。

このとき、申請と承認、差し戻し連絡、再提出、履歴確認が別々のツールに分かれていると、以下のような課題が発生します。

* どの申請が現在どの状態なのか分かりにくい
* 誰が次に対応すべきか分かりにくい
* 差し戻し時に、どの項目を修正すべきか伝わりにくい
* 再提出後に、何が修正されたのか追いにくい
* 承認履歴や操作履歴を後から確認しづらい
* CSV 出力や監査ログ確認が属人化しやすい

私は、業務システムでは「機能があること」だけでなく、実際の運用で迷わず使えることが重要だと考えています。

そのため ReviewFlow では、申請フォーム、承認フロー、差し戻し内容、再提出履歴、申請状態、CSV 出力、監査ログを同じ業務フロー上で扱えるようにしました。

## 4. UI・デザイン方針

ReviewFlow の UI では、見た目の装飾性よりも、業務で迷わず使えることを重視しました。

申請業務では、管理者、承認者、申請者で必要な情報や操作が異なります。
そのため、画面ごとの役割を明確にし、ユーザーが「今、何を確認すべきか」「次に何をすればよいか」を判断しやすい構成を目指しました。

### 1. 操作目的を明確にする

申請者には、申請入力、修正対象の確認、再提出を中心に表示します。
承認者には、申請内容、承認状況、差し戻し履歴、承認・差し戻しボタンを中心に配置します。
管理者には、申請フォーム、承認フロー、提出済み申請、監査ログなど、管理対象ごとに画面を分けて表示します。

ユーザーの役割ごとに必要な操作を整理することで、画面を見たときに「自分は何をすればよいか」が分かることを意識しました。

### 2. 状態を見ただけで分かるようにする

申請中、承認済み、差し戻し、却下などの状態をラベルで表示し、一覧画面や詳細画面で現在の状態を把握しやすくしています。

申請業務では、状態が分からないことが確認漏れや催促の属人化につながります。
そのため、現在の対応者や承認ステップが分かることを重視しました。

### 3. 差し戻し内容を具体的に伝える

差し戻し時には、対象項目とコメントを申請データに紐づけています。

これにより、申請者は「どの項目を、なぜ修正すべきか」を確認しながら再提出できます。
承認者側も、過去の修正依頼や再提出内容を確認しながら審査を進められます。

差し戻しを単なるコメントではなく、業務フロー上の重要な状態として扱った点が、このアプリの特徴です。

### 4. 外部ユーザーでも申請しやすくする

申請者は、必ずしも業務システムに慣れているとは限りません。

そのため、外部申請者向けにはログインアカウントを不要にし、申請に必要な情報入力に集中できる構成にしました。

住民や事業者など、組織外のユーザーでも迷わず申請できるようにすることで、申請者側の利用ハードルを下げることを目指しています。

## 5. 主な機能

### 管理者向け

#### 申請フォーム作成

スペース単位で申請フォームを作成できます。
申請内容に応じてフィールドを追加・並べ替えでき、作成したフォームは公開・アーカイブできます。

#### 承認フロー設定

申請フォームに対して、承認ステップや担当者を設定できます。
複数人での確認や、段階的な承認が必要な申請業務を想定しています。

#### 申請一覧確認

スペース内の提出済み申請を一覧で確認できます。
ステータスや日付をもとに、現在どの申請が対応中なのかを把握できます。

#### CSV 出力

申請データを CSV として出力できます。
集計や内部確認、他システムでの利用を想定し、CSV エクスポートジョブを作成して、完了後にダウンロードできる構成にしています。

#### 監査ログ確認

申請、承認、差し戻し、ユーザー招待、権限変更、申請フォーム作成・編集などの操作履歴を確認できます。
誰が、いつ、何を行ったかを確認できるようにし、問い合わせ対応や内部確認に使える設計にしています。

---

### 申請者向け

#### ログイン不要での申請提出

申請者は、ログインアカウントを持っていなくても申請を提出できます。
外部ユーザーからの申請受付を想定し、アカウント作成の手間を省ける点を特徴としています。

#### 申請フォーム入力

申請フォームに設定された項目に沿って、申請内容を入力できます。
住民・事業者・取引先などの外部ユーザーでも使いやすいよう、必要な項目を順に入力できる構成にしています。

#### 差し戻し内容の確認

申請が差し戻された場合、差し戻し対象の項目、項目別コメント、全体コメント、差し戻し履歴を確認できます。

#### 修正後の再提出

差し戻し後、申請者は修正内容を反映して再提出できます。
不備のある申請を修正し、承認に進めるための流れを扱えるようにしています。

---

### 承認者向け

#### 申請内容確認

申請者が提出した内容、承認進捗、過去の差し戻し履歴を確認できます。

#### 承認

現在の承認ステップに割り当てられた担当者が、申請内容を確認して承認できます。
承認フローに沿って、次のステップまたは完了状態へ進めます。

#### 却下

承認できない申請について、承認フローを終了させる操作として却下できます。

#### 差し戻し

申請内容に不備がある場合、フィールド単位で差し戻し対象を選び、コメント付きで申請者に修正依頼できます。

## 6. 設計上のアピールポイント

### 6-1. 状態遷移を持つ業務ワークフロー

ReviewFlow は、単純な CRUD アプリではなく、申請の状態に応じて操作可能な内容が変わる業務ワークフローとして設計しています。

申請は、下書き、審査中、差し戻し、承認済み、却下といった状態を持ちます。
提出、承認、差し戻し、再提出、却下の操作に応じて状態が遷移します。

特に重視したのは、フォーム送信後の「差し戻し」と「再提出」です。
申請内容に不備があった場合、承認者が修正対象を指定して差し戻し、申請者が内容を修正して再提出できるようにしています。

状態遷移のルールは `ApplicationTransitionPolicy` に集約し、サービス層から直接ステータスを書き換えないようにしました。
これにより、状態変更のルールが分散しない構成にしています。

---

### 6-2. 実務を想定した権限管理

ReviewFlow では、単純な管理者・一般ユーザーというロールだけでなく、申請の状態や承認フロー上の担当者設定に応じて操作権限を判定しています。

たとえば、承認操作は、審査中の申請に対して、現在の承認ステップに割り当てられた担当者、またはテナント管理者のみが実行できるようにしています。

`ApplicationAccessPolicy` では、以下の条件を組み合わせて閲覧・操作権限を判定しています。

* テナント管理者かどうか
* 申請者本人かどうか
* 現在の承認ステップの担当者かどうか
* 過去に承認操作へ関与したユーザーかどうか
* 対象スペースにアクセスできるユーザーかどうか

フロントエンドでも操作ボタンの表示制御を行いますが、最終的な権限判定はバックエンド側で行う構成にしています。

---

### 6-3. 監査ログによる操作履歴の管理

業務システムでは、処理が完了するだけでなく、後から「誰が・いつ・何をしたか」を確認できることが重要です。

ReviewFlow では、申請、承認、差し戻し、却下、ユーザー招待、権限変更、申請フォーム作成・編集などの操作を監査ログとして記録しています。

これにより、問い合わせ対応や内部確認が必要になった場合でも、操作履歴をもとに状況を追跡できます。

---

### 6-4. OpenAPI による型安全な API 連携

ReviewFlow では、フロントエンドとバックエンドの連携ミスを減らすため、OpenAPI を用いた型安全な API 連携を採用しています。

バックエンドでは NestJS の Swagger 連携により OpenAPI スキーマを生成し、API 仕様を管理しています。
フロントエンドでは `openapi-typescript` により TypeScript 型を生成し、`openapi-fetch` のクライアントに適用しています。

API 仕様をフロントエンドとバックエンドの共通契約として扱うことで、リクエストやレスポンスの型ずれを減らし、保守性を高めることを目指しました。

---

### 6-5. バックエンドの責務分離

バックエンドでは、Controller や Service に業務ルールを詰め込みすぎないよう、責務ごとにクラスやサービスを分けています。

| クラス / モジュール                    | 役割                     |
| ------------------------------ | ---------------------- |
| ApplicationAccessPolicy        | 申請の閲覧・レビュー操作が許可されるかを判定 |
| ApplicationTransitionPolicy    | 申請状態の前提チェックと状態遷移       |
| ApplicationFormValueValidator  | フォーム入力値が定義どおりかを検証      |
| ApplicationSubmissionService   | 提出・再提出処理の流れを管理         |
| ApplicationReviewActionService | 承認・却下・差し戻し処理の流れを管理     |
| ApplicationCorrectionService   | 差し戻し対象や修正依頼に関する処理      |
| Repository 層                   | 永続化処理を担当               |

この分離により、仕様変更が発生した場合に、権限、状態遷移、入力検証、永続化のどこを修正すべきかを判断しやすくしています。
